# Anti-Patterns

자주 빠지는 함정의 *유일한 source*. 다른 rules 의 "Anti-Patterns" 절은 이 파일의 해당 앵커로 참조한다.

각 항목은 *행동 가능한 형태*로 표현한다 — "X하지 않는다" 또는 "X 대신 Y한다".

## Code Quality

- 전체 맥락을 읽지 않고 코드 수정
- 기존 유사 코드·유틸리티 탐색 없이 새로 구현
- 실패·경고·엣지 케이스 무시
- Premature optimization / abstraction
- Broad exception handler (`catch (e)` 같은 통짜 캐치)
- Mutation 을 통한 상태 변경 (불변 패턴 위반)
- 깊은 중첩 (>4단계 들여쓰기)
- 하드코딩된 시크릿·값
- 사용되지 않는 변수·import 방치
- 문서·주석에 과거 상태·변경 이력·시행착오를 누적 (현재 상태만 기록 — 이력은 git/PR 이 source)
- AI 에이전트의 추측·수정·롤백 흔적을 결과물에 남기기 ("이전엔 X 였는데 Y 로 변경" 류)

## Surgical / 외과적 변경 위반

- 요청 범위를 넘어선 인접 코드 "개선" (drive-by refactoring)
- 기존 스타일을 본인 선호로 변경 (따옴표 통일, 들여쓰기·줄바꿈, import 정렬)
- 요청에 없는 타입 힌트·docstring·주석 추가
- 사용자가 묻지 않은 새 추상화·플래그·설정 옵션 추가
- 무관한 dead code 함께 삭제 (발견했다면 *언급만* 하고 별도 작업으로 분리)
- 한 PR/커밋에 두 가지 이상 목적 혼합
- "겸사겸사 정리했어요" 류의 변경

## Think Before Coding

- 모호한 요구를 받자마자 코드 작성 시작
- 여러 해석을 머릿속에서 하나로 좁힌 뒤 *알리지 않고* 진행
- "일단 만들어 보고 마음에 안 들면 바꾸시면 됩니다" — 구현 비용을 사용자에게 떠넘김
- 사용자 안이 과복잡해 보여도 푸시백 없이 그대로 구현
- "구현하다 막히면 그때 묻겠다"

## Goal-Driven Execution

- 종료 조건 없이 시작 ("일단 해보고 안 되면 다시")
- "make it work" 류 모호한 지시를 그대로 받아 진행
- 검증 없이 "완료" 보고
- 테스트 없이 "동작 확인됨"이라고 결론
- 단계별 검증 방법을 정하지 않고 다단계 작업 시작

## Problem Solving

- 근본 원인 파악 없이 빠른 수정
- 증상 치료(try-catch 도배)로 원인 회피
- 첫 번째 그럴듯한 설명에서 멈춤
- 원인 파악 없이 재시도·자동 재시작만 추가
- 수정 후 검증 생략
- "일시적 이슈"로 단정하고 종결
- 같은 근본 원인이 다른 코드 경로에 있는지 확인 안 함

## Testing

- 통합 테스트를 전부 mock 으로 대체 (프로덕션 동작과 괴리)
- 버그 수정 시 회귀 테스트 누락
- 테스트 간 상태 공유로 순서 의존성 발생
- 실패하는 테스트를 수정 대신 skip / `.only` 처리
- 테스트가 잘못된 게 아닌데 *테스트*를 수정해 통과시킴

## Git & Deployment

- 사용자 허가 없이 `commit` 또는 `push`
- 훅·서명 우회 (`--no-verify`, `--no-gpg-sign`)
- 공유 브랜치에 force push
- 시크릿·바이너리·생성 파일 커밋
- 덮어쓰기 전 백업·검증 생략
- `rm -rf`, `git reset --hard` 등 파괴적 명령을 확인 없이 실행
- 최신 커밋만 보고 PR 본문 작성 (전체 이력 분석 누락)

## Communication & Process

- 사용자 요구 없이 자동 커밋/리팩토링
- 계획(Plan Mode) 없이 대규모 변경 시작
- 불확실한 내용을 질문 없이 추측으로 작성
- 긴 요약·장황한 설명 남발 (사용자는 diff 를 볼 수 있다)
- 없는 Skill·Agent 를 추측해서 호출
- 훈련 데이터를 맹신하고 최신 문서(Context7 등) 확인 생략

## Claude Code 고유

- `EnterPlanMode` 없이 다중 파일 변경 시작
- 독립적 도구 호출을 순차 실행 (병렬 가능한데 직렬)
- 대규모 탐색을 subagent 없이 메인 컨텍스트에서 수행
- `TaskCreate` 없이 3단계 이상 복잡 작업 추적
- Context7 에서 조회해야 할 라이브러리 문서를 훈련 데이터로 답변
- 컨텍스트 마지막 20% 에서 대규모 리팩토링 시도
- 이미 실행 중인 스킬 재호출

## Working If / 잘 작동하고 있다는 신호

이 가이드라인이 실제로 효과가 있는지 자가 점검하는 척도. 작업 종료 시 또는 PR 리뷰 시 확인.

### 좋은 신호 (Working)

- diff 에 *요청과 직접 무관한* 변경이 없다 (스타일 통일·인접 리팩토링·dead code 제거가 섞이지 않음)
- 과복잡화로 인한 재작성·롤백이 줄어든다
- 명확화 질문이 *구현 후 실수* 가 아니라 *구현 전*에 발생한다
- PR 이 작고 단일 목적이다 (요청 1개 = PR 1개)
- 사용자 요청 분량 대비 변경 라인 수가 비례적이다 (한 줄 버그 수정에 200줄 diff = 경고)
- 종료 조건이 테스트·측정·lint 로 자동 검증 가능하다

### 나쁜 신호 (다시 점검 필요)

- "겸사겸사 정리했어요" 류의 PR
- "테스트 없이 수동으로 동작 확인 끝" 류의 완료 보고
- 사용자가 묻지 않은 새 추상화·플래그·설정 옵션이 추가됨
- 같은 기능이 한 PR 안에 두 번 이상 다른 방식으로 구현됨
- 따옴표·들여쓰기·import 순서 변경이 다른 변경에 섞여 있음
- 모호한 요구를 받았는데 질문 없이 바로 구현이 시작됨
- "일단 해보고 안 되면 다시" — 종료 조건 없이 시작
- 200줄 작성 후 "더 단순하게 다시 짤 수 있었음"을 깨달음

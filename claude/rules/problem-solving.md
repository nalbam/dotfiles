# Problem Solving & Root Cause Analysis

CLAUDE.md `## Before Changing Code`, `## Problem Solving`, `## Goal-Driven Execution` 의 *유일한 상세 source*.

증상을 덮지 말고 근본 원인을 찾는다. 명령형 지시는 검증 가능한 종료 조건으로 변환한 뒤 시작한다.

## Think Before Coding / 코딩 전 사고

**원칙: 가정을 추측으로 통과시키지 않는다. 혼란을 숨기지 않는다. 트레이드오프를 표면화한다.**

LLM 이 가장 자주 빠지는 함정은 *조용히 하나의 해석을 골라 진행* 하는 것이다. 잘못된 가정으로 만든 코드는 50줄짜리든 500줄이든 전부 폐기된다.

### MUST (구현 시작 전)

- **가정을 명시한다** — "X 라고 가정하고 진행한다" 를 한 줄이라도 적는다.
- **다중 해석을 표면화한다** — 요구사항이 모호하면 가능한 해석 2~3개를 나열하고 선택을 요청한다. 조용히 하나를 고르지 않는다.
- **단순한 대안을 먼저 제시한다** — 더 간단한 접근이 보이면 즉시 말한다. 사용자 안이 과복잡해 보이면 근거와 함께 푸시백한다.
- **혼란이 생기면 멈춘다** — "구현하다 막히면 다시 묻겠다" 는 금지. 무엇이 불명확한지 *이름 붙여* 질문한다.

### 모호함 해소 형식 (예시)

```
요청: "검색을 더 빠르게 해줘"

다음 중 어떤 의미인지 확인 부탁드립니다:
1. 응답 시간 단축 (예: p95 500ms → 100ms) — 인덱스·캐시
2. 동시 처리량 증가 — async·커넥션 풀
3. 체감 속도 개선 — 부분 결과·점진적 로딩

현재 baseline 은 [측정값]입니다. 가장 중요한 축이 무엇인가요?
```

## Goal-Driven Execution / 목표 기반 실행

**원칙: 명령형 지시를 *검증 가능한 종료 조건* 으로 먼저 변환한 뒤 실행한다.**

약한 기준("그냥 동작하게")은 매 단계마다 사용자 확인이 필요하다. 강한 기준은 LLM 이 독립적으로 루프를 돌아 스스로 완료를 판정할 수 있게 한다.

### 변환 규칙

| 모호한 지시 | 검증 가능한 목표 |
|------------|------------------|
| "Add validation" | 잘못된 입력에 대한 실패 테스트 작성 → 구현 → 통과 확인 |
| "Fix the bug" | 버그를 재현하는 테스트 작성 → 통과시킴 → 회귀 테스트로 잠금 |
| "Refactor X" | 리팩토링 전후로 동일 테스트 스위트가 통과 |
| "Improve performance" | 측정 baseline + 목표 수치 정의 (예: p95 < 100ms) → 측정 → 검증 |
| "Make it cleaner" | 정의된 lint/typecheck/coverage 기준 충족 |
| "Make it work" | "무엇이 어떻게 되면 끝인지" 사용자에게 먼저 확인 |

### 다단계 작업 계획 형식

각 단계는 *독립적으로 검증 가능* 해야 한다. 검증 방법이 정해지지 않은 단계는 계획에 넣지 않는다.

```
1. [단계] → 검증: [구체적 확인 방법]
2. [단계] → 검증: [구체적 확인 방법]
3. [단계] → 검증: [구체적 확인 방법]
```

### 시작 전 자가 점검

- [ ] 종료 조건이 한 문장으로 표현되는가
- [ ] 종료 여부를 자동으로 확인할 수 있는가 (테스트·측정·lint)
- [ ] 사용자 추가 입력 없이 단계별 진행이 가능한가

위 셋 중 하나라도 No → 구현 전 사용자에게 종료 조건 확인.

## Investigation Steps / 조사 단계

1. **Reproduce** — 에러 메시지·스택 트레이스·환경 정보를 기록
2. **Investigate**
   - 에러 메시지를 정독한다 (종종 근본 원인이 그대로 담겨 있음)
   - 실패 직전 로그의 패턴·경고·에러 탐색
   - 진입점부터 실패 지점까지 실행 경로 추적
   - 최소 실험으로 가정 검증
3. **Root Cause**
   - "Why?" 를 근본 이슈에 도달할 때까지 반복 (보통 5회)
   - 증상과 원인을 구분 (예: "API 가 500 반환" vs. "커넥션 풀 고갈")
   - 일회성 실패가 아닌 시스템 이슈인지 확인
4. **Fix** — 근본 원인 수정. 워크어라운드는 *임시* 임을 명시 + 후속 작업 등록.
5. **Verify** — 회귀 테스트 추가. 같은 원인이 다른 코드 경로에 있는지 스캔.

## Example

**Symptom**: 프로덕션 간헐적 크래시

- ❌ Bad: 에러 핸들링 추가 + 자동 재시작
- ✅ Good: 프로파일링 → 임시 파일 미정리로 인한 메모리 누수 식별 → cleanup 로직 + 제한값 → 부하 테스트로 검증

## 핵심 원칙

- 첫 번째로 떠오르는 그럴듯한 설명에서 멈추지 않는다
- 재현 → 가설 → 실험 루프를 빠르게 돈다
- 수정 후 반드시 회귀 테스트 추가
- 같은 근본 원인이 다른 코드 경로에 있는지 스캔

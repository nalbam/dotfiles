# Problem Solving & Root Cause Analysis

버그·장애 해결 시 따를 체계적 절차. 증상을 덮지 말고 근본 원인을 찾는다.

## Investigation Steps

1. **Reproduce** — 에러 메시지, 스택 트레이스, 환경 정보를 기록
2. **Investigate**
   - 에러 메시지를 주의 깊게 읽는다 (종종 근본 원인이 그대로 담겨 있음)
   - 로그에서 실패 직전의 패턴·경고·에러 탐색
   - 진입점부터 실패 지점까지 실행 경로 추적
   - 최소 실험으로 가정 검증
3. **Root Cause**
   - "Why?"를 근본 이슈에 도달할 때까지 반복 (보통 5회)
   - 증상과 원인을 구분 (예: "API가 500 반환" vs. "커넥션 풀 고갈")
   - 일회성 실패가 아닌 시스템 이슈인지 확인
4. **Fix** — 근본 원인 수정. 필요하면 워크어라운드를 문서화 (단, 임시 조치임을 명시)
5. **Verify** — 철저히 테스트, 회귀 테스트 추가, 유사한 이슈가 다른 곳에도 있는지 점검

## Example

**Symptom**: Intermittent crashes in production

- **Bad**: 에러 핸들링 추가 + 자동 재시작
- **Good**: 프로파일링 → 임시 파일 미정리로 인한 메모리 누수 식별 → cleanup 로직 추가 + 제한값 도입 → 부하 테스트로 검증

## 핵심 원칙

- 첫 번째로 떠오르는 그럴듯한 설명에서 멈추지 않는다
- 재현 → 가설 → 실험 루프를 빠르게 돈다
- 수정 후 반드시 회귀 테스트 추가
- 같은 근본 원인이 다른 코드 경로에도 있는지 스캔

## Anti-Patterns (이 주제에 특화)

- try-catch 도배로 증상 숨기기
- 원인 파악 없이 재시도·자동 재시작만 추가
- 수정 후 검증 생략
- "일시적 이슈인 듯"으로 단정하고 종결
